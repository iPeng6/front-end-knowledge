踩坑-reactnative fetch self-signed https

编程
  • ReactNative

(react native 请求自签名的无效证书的https)
我们知道iOS9之后苹果默认启用https验证了，这就要我们的服务器做https升级，当然我们可以通过plist配置来允许http访问，可势必会有一些安全问题，既然要启用https可是https的证书是非常贵的，我们测试开发的时候就随便用了个自己签名的证书，于是问题就来了

当使用ReactNative的时候请求网络都是用的fetch，但是从调试错误信息来看，fetch会解释成iOS本地的NSURLConnection/NSURLSession来做网络访问，而iOS默认这些方法都会去验证证书，而我们明知道这个证书是无效的，始终TypeError: Network request failed，然后我尝试用原生的写一遍，提示

Error Domain=NSURLErrorDomain Code=-1202 “The certificate for this server is invalid. You might be connecting to a server that is pretending to be “your.domain.com” which could put your confidential information at risk.”

明显是说证书无效，后来查了很多关于这个问题的解决方法比如stackoverflow上的一个总结回答Https iOS with self signed certificate

他也说了

By default, Cocoa refuses all SSL connections when the certificate is invalid…

默认cocoa框架拒绝所有无效证书的SSL连接访问，通过AFN我们也能通过代理来绕过验证

AFSSLPinningModeNone这个枚举还有另外的几个值

网上也有很多方法来简单验证一下这个无效的自签名证书来保证即使用的是无效证书也能验证一下这个证书跟服务器的一致性来稍微提高一下安全性，具体后面有需要再研究

我们这里是react native，用的是fetch，转成oc会默认验证，就会默认失败，如果自己改造代码封装一套给React Native 调用势必很麻烦，有没有一种不动代码就能更好的开发调试无效证书的https的方法呢？

最后通过代理的方式解决了，折腾了好久

通过Fiddler这个windows下的软件可以忽略https证书一种https中间人攻击的方式

Tools->Fiddler Options->HTTPS->勾选Decrypt HTTPS traffic & Ignore server certificate errors(unsafe)

Connections-> Allow remote computers to connect,这里的端口默认是8888

这个是运行在windows上的，查一下电脑ip，然后我的mac设备，iPhone设备的网络代理都设置成这台电脑，然后所有的连接都是通过这台电脑了，然后我们的设备浏览器访问一下这台机器的地址 http://ip:8888 会有个页面，最后一个链接，下载证书FiddlerRoot certificate，双击安装证书mac的钥匙串就有有这个证书双击全部信任，至此我们的电脑就可以访问所有的https了，我们跟fiddler代理建立了一个信任关系，然后fiddler帮我们忽略其他服务器的证书验证

mac设置代理
网络偏好设置->wifi高级->代理->安全Web代理(https)设置ip端口->好->应用

后来想window上的Fiddler对应mac肯定也有类似的工具，那就是charles，charles也可以有类似配置，打开charles
Help->SSL Proxying-> Install… 都装上分别是机器、模拟器、真机看着提示很容易，然后也一切OK了（–!我竟然饶了一大圈）

这里有两个注意点

  1. wifi代理如果是代理的https的要勾选安全Web代理(https)
  2. charles的证书安装最好通过软件的Help安装，网站上下载的证书他也说了3.10以上可能无效does not work
  3. charles监听要把要监听的地址添加到SSL Proxying setting的列表里，这一点没有fiddler好使

其实以上这些坑踩完还没结束，安卓端也要兼顾，发现android上有个坑就是，手机上虽然安装了证书也设置了代理服务器，但是只有浏览器和谷歌服务软件走代理，而自己安装的App可能不走这个代理，如果代理不上那以上工作设置都是白忙活了

最终我们打算还是从代码层面去忽略证书吧，好心酸
